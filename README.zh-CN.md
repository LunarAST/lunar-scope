# lunar-scope — 生态可视化画布
**面向 LunarAST 协议体系的静态、可离线运行拓扑渲染工具**

## 1. 用途与运行边界
`lunar-scope` 是 LunarAST 生态的可视化层。它读取标准格式的 `lunar-map.json` 文件，渲染交互式多层拓扑图，展示微服务接口、契约对齐状态以及生态异常问题。

### 1.1 核心功能
- 解析 `lunar-map.json`，渲染项目节点与契约对齐连线
- 支持连线聚合：将同一组项目间的多条调用关系合并为单条连线
- 展示生态异常报告：包含未使用接口、孤立消费端、跨层违规项
- 无权限引导式同步：为异常契约生成 YAML 补丁，支持一键复制，可在本地命令行工具中执行
- 打包为单 HTML 文件，可完全离线运行，无任何运行时依赖

### 1.2 功能边界（本组件不负责）
- 不会向磁盘写入任何文件，无需文件读写权限
- 不依赖后端服务
- 不执行契约对齐计算（该能力由 `lunar` 工具提供）
- 不处理身份认证与授权（该能力由 `lunar-gateway` 网关提供）

## 2. 数据规约
`lunar-scope` 读取 `lunar-map.json` 文件，该文件需严格遵循 `schemas/lunar-map.schema.json` 定义的数据结构。

### 2.1 必要字段说明
- `version`：数据规约版本号（示例：`"0.5.0"`）
- `projects`：项目节点数组，每个节点包含 `name`、`type`、`sha`、`scanStatus`、`interfaces` 字段
- `alignments`：单条契约对齐记录数组
- `aggregatedEdges`：聚合连线数组（按客户端-服务端分组）
- `anomalies`：异常信息对象，包含 `unusedEndpoints`、`orphanedConsumers`、`crossLayerViolations` 三类异常

### 2.2 路径序列化规则
所有接口路径统一遵循标准格式：普通文本段使用 `/{segment}`、路径参数段使用 `/{paramName}`、通配符段使用 `/*`。

## 3. 视觉与交互规范
### 3.1 布局规则
- 基于 `elkjs`（Eclipse 布局引擎）实现确定性分层布局
- 服务节点从左至右排布：左侧为调用方（客户端），右侧为被调用方（服务端）
- 无物理抖动效果，节点位置可完全复现

### 3.2 节点渲染规则
- 背景采用深炭黑色（`#0B0B0C`），搭配浅灰色网格线
- 项目以矩形卡片形式展示：左侧为**对外接口端口**，右侧为**调用接口端口**
- 节点边框颜色区分项目类型：服务端（绿色）、客户端（蓝色）、混合类型（琥珀色）
- 接口端口以彩色圆点展示：契约正常对齐为实心圆点；未使用/孤立接口为空心圆点，并保留请求方法对应的填充色

### 3.3 按契约状态区分连线与端口样式

| 状态 | 视觉样式 | 交互规则 |
|:---|:---|:---|
| `Aligned`（已对齐）| 细绿色实线（`#10B981`） | 无交互，代表契约状态正常 |
| `ParamNameMismatch`（参数名不匹配）| 蓝色虚线（`#3B82F6`） | 仅诊断告警，无操作入口；悬浮可查看参数名差异 |
| `Orphaned`（孤立接口）| **悬空线缆样式**（`#F59E0B`） | 支持交互。从调用端端口延伸出黄色悬空连线与问号连接器，可拖拽至目标服务端端口完成磁吸对齐，并自动生成 YAML 补丁 |
| `MethodMismatch`（请求方法不匹配）| 粗红色实线（`#EF4444`） | 点击唤起侧边面板，展示请求方法冲突详情 |
| `Unverified`（未校验）| 浅灰色虚线（`#6B7280`） | 悬浮提示：因 CI 扫描失败，数据源不可用 |
| `Unused`（未使用，端口级）| 石板灰色对外端口（`#4B5563`） | 直接在对外端口上展示未使用标记，无连线延伸 |

#### 3.3.1 端口视觉设计规范
`lunar-scope` 中所有接口端口均采用三层视觉编码体系，样式完全通过 CSS 自定义属性解耦。

| 状态 | 填充色 | 边框色 | 外发光 | 对外端口形状 | 调用端口形状 |
|:---|:---|:---|:---|:---|:---|
| `aligned`（已对齐） | 请求方法对应实色 | 请求方法对应色 | 无 | 圆形 | 圆形 |
| `unused`（未使用） | 画布背景色（空心） | 请求方法对应色 | 红色（`#EF4444`） | 圆形 | 无 |
| `orphaned`（孤立） | 画布背景色（空心） | 请求方法对应色 | 黄色（`#F59E0B`） | 无 | 菱形 |
| `mismatch`（不匹配） | 请求方法对应实色 | 请求方法对应色 | 红色（`#EF4444`） | 圆形 | 圆形 |

**层级定义说明：**
- **填充色**：契约正常对齐时，使用请求方法对应实色；未使用/孤立接口则沿用画布背景色，呈现空心效果。
- **边框色**：统一使用请求方法对应色，保留 HTTP 请求方法的视觉标识（GET 绿色、POST 蓝色、DELETE 红色等）。
- **外发光（阴影）**：正常对齐无外发光；未使用端口添加红色外发光；孤立端口添加黄色外发光。在不遮挡请求方法标识的前提下，快速区分端口状态。
- **形状**：所有对外端口、已对齐的调用端口均为圆形；孤立状态的调用端口使用菱形（圆角 30%），直观体现“连接异常”。

**实现要求**：所有颜色值统一引用 CSS 自定义属性（例如 `var(--lunar-method-get-text)`），配置集中存放于 `src/portStyles.ts`。保证项目节点、卡片标识、端口样式使用同一套数据源，视觉表现完全一致。

### 3.4 连线聚合规则
- 同一组客户端与服务端之间的多条契约关系，合并为单条聚合连线
- 连线标签展示调用次数（示例：`12 Calls`）
- 点击聚合连线，唤起仿 Vercel 风格侧边面板，展示全部独立接口路径、请求方法与契约状态

### 3.5 端口级连线路由（规划版本 v0.7.0）
当 `aggregatedEdges` 数组内包含 `ports` 子数组时，每条端口级连接会从指定源连接柄渲染至目标连接柄。

- **源连接柄**：客户端节点中，下标为 `sourcePortIndex` 的调用端口
- **目标连接柄**：服务端节点中，下标为 `targetPortIndex` 的对外端口
- 连线样式以**单端口自身状态**为准，不受聚合状态影响
- 若 `ports` 字段缺失或为空，自动降级为聚合连线渲染（兼容旧版本）

该能力可实现：
- 接口与连线精准绑定
- 单端口独立状态展示（正常对齐显示实线、孤立接口显示悬空连线、未使用接口无连线）
- 后续支持孤立端口的拖拽磁吸对齐功能

### 3.6 生态异常面板
- 顶部工具栏可唤起独立异常面板
- 汇总 `anomalies` 字段内的**未使用接口**、**孤立消费端**、**跨层违规项**
- 点击任意异常项，画布自动高亮对应的节点与连线

### 3.7 无权限引导式同步（悬空线缆补丁功能）
1. 用户拖拽黄色的 `Orphaned`（孤立）调用端悬空连线，靠近服务端正常对外端口
2. 系统触发磁吸对齐，并自动计算可修复 `.lunar/interfaces.yml` 的补丁内容
3. 底部控制栏展示 YAML 差异内容，并提供两个复制按钮：
   - **复制补丁**：复制 YAML 补丁代码片段
   - **复制命令**：复制 `lunar sync --apply` 执行命令
4. 用户在本地粘贴并执行命令，由 `lunar` 命令行工具自动完成文件备份与写入

### 3.8 AI 辅助契约维护
本地或云端 AI 工具可在**不获取设备读写权限**的前提下，协助维护生态契约。

1. **AI 获取上下文**：AI 读取 `lunar-map.json`（通过 `lunar-serve` 服务或共享链接），识别缺失、异常或不完整的契约声明
2. **生成修复建议**：AI 输出符合 `interfaces.yml` 格式规范的 YAML 补丁文件，开发者将文件保存至 `.lunar/suggestions/[自定义名称].yaml`
3. **审核与合并**：开发者执行 `lunar sync --apply`，命令行工具会依次完成以下操作：
   - 列出所有待处理的补丁建议，并预览内容差异
   - 等待用户确认执行
   - 将当前 `interfaces.yml` 备份至 `.lunar/.backup/` 目录
   - 以字段增量覆盖的方式合并补丁内容
   - 将已处理的补丁文件移动至 `.lunar/suggestions/merged/` 目录
4. **提交与同步**：开发者将更新后的 `interfaces.yml` 与归档补丁提交至 Git。下次执行 `lunar map` 时，新的契约意图会自动生效，供后续所有人工或 AI 工具使用。

该工作流全程不提升权限、操作可追溯，持续优化整个生态的契约规范。

## 4. 技术栈

| 分层 | 技术选型 | 选用原因 |
|:---|:---|:---|
| 前端框架 | React + Vite | 生态成熟、热模块替换（HMR）速度快、原生支持 TypeScript |
| 图形库 | xyflow（React Flow） | 原生支持节点与端口架构、内置视口虚拟化能力 |
| 布局引擎 | elkjs | 布局结果可复现、无随机偏移，分层布局能力优秀 |
| 打包工具 | vite-plugin-singlefile | 将所有资源内联，输出单文件离线 HTML |
| 开发语言 | TypeScript | 为复杂数据结构提供强类型约束 |

## 5. 部署与数据注入规范
### 5.1 离线模式（数据注入占位符）
1. **Vite 编译阶段**：通过 `vite-plugin-singlefile` 编译 React 项目，生成模板文件，并内置标准数据占位符：
   `<script>window.lunarMapPayload = /* LUNAR_MAP_DATA_PLACEHOLDER */ null;</script>`
2. **数据注入阶段**：在上传至 S3/R2 对象存储 或 本地命令行编译时，`lunar` 工具通过文本替换，将 JSON 数据填入占位符，最终生成独立完整的单文件 `route-scope.html`
3. **权限说明**：可在完全物理隔离、无网络的环境中安全运行

## 6. 与其他组件的关联关系

| 组件 | 关联说明 |
|:---|:---|
| `lunar` 命令行工具 | 生成 `lunar-map.json`，为本组件提供数据源 |
| `lunar-gateway` 网关 | 在线环境下，托管 `lunar-map.json` 与本组件静态资源 |
| RouteAST 子协议 | 定义本组件所展示的各类契约状态枚举 |
| LunarAST 生态总规范 | 定义四层架构体系与数据生成、同步机制 |

`lunar-scope` 仅作为**数据消费方**，不会写入本地文件、修改任何配置。
